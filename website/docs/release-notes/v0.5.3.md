# Gitmoot v0.5.3

A fail-closed safety release: an agent that can `implement` but has no write
policy is now refused early with actionable guidance instead of running a whole
job and writing nothing.

## What's new since v0.5.2

### Implement write-policy fail-closed (behavior change)

- **feat(cli,workflow): refuse implement + non-write policy** (#452) â€” an agent (or
  ephemeral worker) that carries the `implement` capability while its autonomy
  policy grants no headless write (`auto`/empty or `read-only`) is now **refused**
  rather than silently producing no files. The same actionable message fires at
  four seams: `gitmoot agent start`, `gitmoot agent subscribe`, implement-job
  **dispatch** (which catches pre-existing agents and later policy edits), and
  ephemeral delegation specs (ties #451).

  **Why this is a behavior change:** the default `auto` policy emits no
  `--permission-mode`, so a headless `claude -p` / `codex exec` run inherited its
  write capability from whatever ambient Claude config existed on the host â€”
  non-deterministic across environments (it wrote where permissive settings
  existed and silently denied where they didn't). Existing `auto`+`implement`
  agents were already unreliable; failing closed is strictly an improvement.

  **The fix when you hit the refusal:** set `--policy danger-full-access` for full
  headless implementation (file writes plus `go`/`git`/`gh` via Bash), or
  `--policy workspace-write` for edits-only. Note `workspace-write` maps to Claude
  `acceptEdits`, which auto-accepts file edits but does **not** unblock Bash, so
  `go`/`git`/`gh` stay blocked â€” use `danger-full-access` for a full implement
  job. `read-only`/`ask`/`review` agents are unaffected.

- **docs: policy â†’ permission-mode mapping** (#452) â€” `SAFETY.md`, the CLI
  reference, and the result-contract reference now document the full
  policyâ†’`--permission-mode` table (`read-only`â†’`plan`,
  `workspace-write`â†’`acceptEdits`, `danger-full-access`â†’`bypassPermissions`,
  `auto`â†’no flag), the ambient-config non-determinism of `auto`, and that
  `acceptEdits` does not unblock Bash.

### Follow-up (deferred)

- A runtime **denied-write classifier** (a structured `permission_denials` signal,
  sibling to the auth/session classifiers) was deferred: a real non-empty
  `permission_denials` sample could not be captured (headless writes succeeded on
  the validation host), so its shape is unknown and not guessed.
